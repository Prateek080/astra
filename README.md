# Astra 🏹 (अस्त्र)

**A plugin for the complete development lifecycle.**

Works with [Claude Code](https://claude.ai/code) and [Cursor](https://cursor.com). Same commands, agents, and skills — one plugin, both editors.

```
Idea → Spec → Plan → Implement → Review → Ship
```

Cross-project. Stack-agnostic. Works in any codebase.

---

## Prerequisites

| Requirement | Why |
|---|---|
| [Claude Code](https://claude.ai/code) or [Cursor](https://cursor.com) | Astra works as a plugin for both |
| `git` | Cloning Astra, version control in `/astra:ship` |
| [`gh` CLI](https://cli.github.com/) | PR creation in `/astra:ship` |
| `node` / `npx` | MCP servers (Context7, Playwright, DeepWiki) |

## Installation

```bash
git clone git@github.com:Prateek080/astra.git ~/astra
```

**Enable globally** *(Astra loads in every project, every session)*

```bash
# zsh
echo 'alias claude="claude --plugin-dir ~/astra"' >> ~/.zshrc && source ~/.zshrc

# bash
echo 'alias claude="claude --plugin-dir ~/astra"' >> ~/.bashrc && source ~/.bashrc
```

**Or enable for a single project** *(Astra loads only when you pass the flag)*

```bash
cd /path/to/your/project
claude --plugin-dir ~/astra
```

**Verify:** Run `/help` inside Claude Code — you should see all `/astra:*` commands.

> **Using Cursor?** See [Cursor setup guide](docs/cursor.md) for Cursor-specific installation.
>
> **For teams:** Each developer clones Astra and picks their editor. The project-specific files generated by `/astra:init` (`CLAUDE.md`, `.claude/rules/`, `.cursor/rules/`) get committed to git — that's how the team shares project context.

## Setup

### `/astra:setup` — Configure your machine *(run once)*

```
/astra:setup
```

Interviews you about your stack, conventions, and preferences, then configures your global environment:

| What it configures | Why |
|---|---|
| `~/.claude/settings.json` | Sets effort to high (deeper thinking) + auto-compact at 80% (prevents context from running out) |
| `~/.claude/CLAUDE.md` | Your personal coding standards — loaded into every session across all projects |
| `/permissions`, `/sandbox`, `/statusline` | Walks you through these interactive commands step by step *(Claude Code only — Cursor manages these in Settings UI)* |

### `/astra:init` — Set up your project *(run once per codebase)*

```
/astra:init
```

Scans your project (README, configs, directory structure), asks 2-4 clarifying questions, then generates:

| What it generates | Purpose |
|---|---|
| `CLAUDE.md` | Project summary — stack, structure, dev commands, conventions. Loaded every session. |
| `.claude/rules/backend.md` | Backend conventions — *"follow route pattern in `app/api/users.py`, use ORM models, add tests"* |
| `.claude/rules/frontend.md` | Frontend conventions — *"reuse `components/ui/`, follow existing data fetching patterns"* |

Rules reference real files from *your* codebase, not generic advice. Only creates rules for directories that exist.

### What files does Astra create in your project?

| File | Created by | Commit to git? |
|---|---|---|
| `CLAUDE.md` | `/astra:init` | **Yes** — team shares project context |
| `.claude/rules/*.md` | `/astra:init` | **Yes** — Claude Code conventions |
| `.cursor/rules/*.mdc` | `/astra:init` | **Yes** — Cursor conventions |
| `SPEC.md` | `/astra:spec` | Optional — useful as documentation |
| `PLAN.md` | `/astra:plan` | Optional — useful for tracking |
| `docs/solutions/*.md` | `/astra:compound` | Optional — project knowledge base |
| `docs/.agent-memory/*.md` | Agents (auto) | **No** — personal learnings, add to `.gitignore` |

**You're ready to build.**

## Choosing the Right Workflow

Not every task needs the full workflow. Match the commands to the size of the task.

| Size | Example | Workflow |
|---|---|---|
| **Tiny** | Typo, rename, config change | Just ask Claude directly |
| **Small** | Single-file fix, add a field | Just ask Claude → optionally `review` |
| **Medium** | Feature across 2-5 files | `plan` → `implement` → `review` → `ship` |
| **Large** | New feature, unclear scope | `spec` → fresh context → `plan` → `implement` → `review` → `ship` |
| **Cleanup** | Refactor, optimize, pay debt | `debt-audit` → `plan` → `implement` → `review` → `ship` |

> All commands are prefixed with `/astra:` — e.g., `/astra:plan`

## Command Reference

### `/astra:spec` — Define what to build

| | |
|---|---|
| **Input** | Feature description (optional) |
| **Output** | `SPEC.md` |
| **Next** | Fresh context → `/astra:plan` |

```
/astra:spec "user auth with Google and GitHub OAuth"
/astra:spec
```

Interviews you about the feature — problem, users, scope, constraints, edge cases, non-requirements. Challenges your assumptions and digs into the hard parts. Produces a structured spec with requirements, non-requirements, and open questions.

---

### `/astra:plan` — Break the work into phases

| | |
|---|---|
| **Input** | `SPEC.md` or inline description |
| **Output** | `PLAN.md` |
| **Next** | `/astra:implement` |

```
/astra:plan
/astra:plan "add pagination to the transactions API"
```

Delegates codebase exploration to a subagent (keeps your context clean). Identifies existing patterns, files to modify vs create, and test infrastructure. Produces a phased plan where each phase has scope, tasks, a test gate, and a parallel flag.

---

### `/astra:implement` — Build phase by phase

| | |
|---|---|
| **Input** | `PLAN.md` or phase number |
| **Output** | Working code with passing tests |
| **Next** | `/astra:review` |

```
/astra:implement
/astra:implement "phase 3"
```

Executes phases from the plan. Before writing new code, searches the codebase for existing patterns to reuse. Runs the test gate after each phase — won't proceed until it passes. Phases marked `Parallel: yes` are automatically grouped and executed concurrently via parallel subagents. Suggests clearing context between phase groups. Stops and asks after 2 failed attempts instead of guessing.

---

### `/astra:review` — Catch issues before shipping

| | |
|---|---|
| **Input** | Uncommitted changes, file path, or git ref |
| **Output** | Findings by severity (Critical / Warning / Suggestion) |
| **Next** | Fix criticals → `/astra:ship` |

```
/astra:review
/astra:review src/auth/
/astra:review main..HEAD
```

Delegates to a reviewer subagent with a **50+ item checklist** covering security, performance, quality, reusability, tech debt, accessibility, and API design. Each finding includes the file, line number, what's wrong, and how to fix it.

---

### `/astra:ship` — Commit and open a PR

| | |
|---|---|
| **Input** | Base branch (optional, defaults to main) |
| **Output** | PR URL |

```
/astra:ship
/astra:ship develop
```

Runs pre-flight checks before committing:

| Check | What it catches |
|---|---|
| Test suite | Failing tests |
| Linter/formatter | Style violations |
| Secrets scan | API keys, credentials, `.env` files in diff |
| Debug artifacts | `console.log`, `print()`, `debugger` statements |

Stages files explicitly (never `git add .`), commits with a clean imperative message, pushes, and opens a PR via `gh`.

---

### `/astra:debug` — Find and fix the root cause

| | |
|---|---|
| **Input** | Error message or description |
| **Output** | Root cause analysis + verified fix + regression test |
| **Next** | `/astra:review` → `/astra:ship` |

```
/astra:debug "TypeError: cannot read property 'id' of undefined"
/astra:debug
```

Delegates to a debugger subagent. Forms hypotheses, tests them systematically (no guess-and-check). Implements the smallest fix that addresses the root cause, writes a regression test, and reports: root cause, evidence, what was changed, and how to prevent similar bugs.

## Additional Skills

| Skill | What it does | When to use |
|---|---|---|
| **`/astra:compound`** | Documents a solved problem to `docs/solutions/` with searchable categorization. Builds a project knowledge base over time. | After fixing a non-trivial bug or making an architectural decision |
| **`/astra:debt-audit`** | Scans for dead code, orphaned TODOs, deprecated patterns, missing tests, duplication. Outputs a prioritized report. | Before a refactor or cleanup sprint |
| **`/astra:workflow-guide`** | Explains which commands to use for new projects, new features, and optimizations. | When you're unsure which workflow to follow |
| **`/astra:retrospective`** | Compares spec and plan to what was actually built. Suggests improvements for your workflow. | After shipping a feature |

## Context Management

> **Why this matters:** AI coding assistants have a finite context window. As the conversation grows, older instructions get compressed or lost. When context fills up, the assistant forgets your conventions, misses details, and quality drops. **Managing context is the single most important habit for getting good results.**

| Situation | Claude Code | Cursor |
|---|---|---|
| Finished a task, starting something new | `/clear` | Start a new chat |
| Between implementation phases | `/clear` | Start a new chat |
| Same mistake keeps repeating | `/clear` + rephrase | New chat + rephrase |
| Long session, can't clear yet | `/compact` (do this around 50% usage) | Context usage visible in UI — start a new chat when heavy |
| Want to monitor usage | `/statusline` (set up once during `/astra:setup`) | Context usage visible in Cursor UI |

**Built-in protections:**
- Astra commands delegate heavy exploration to **subagents** — their file reads happen in a separate context, not yours.
- Auto-compaction set to **80%** during `/astra:setup` (default 95% is too late).

## Under the Hood

### Agents

Commands delegate to specialized agents, each running in its own context window.

| Agent | Role | Tools | Learns |
|---|---|---|---|
| **planner** | Explores codebase, creates plans | Read-only | Yes |
| **implementer** | Writes code, runs tests | Read + Write | Yes |
| **reviewer** | 50+ item review checklist | Read-only | Yes |
| **debugger** | Root cause analysis and fix | Read + Write | Yes |

Learning agents accumulate knowledge across all your projects — the reviewer remembers recurring issues, the debugger remembers common root causes. Memory persists natively in Claude Code. Cursor users can add persistence via an MCP memory server (see [Cursor guide](docs/cursor.md#agent-memory-on-cursor)). The implement command automatically detects parallel phases in `PLAN.md` and spawns subagents to execute them concurrently — both editors support worktree-based parallel execution.

### MCP Servers *(auto-loaded)*

| Server | Purpose |
|---|---|
| **Context7** | Up-to-date library docs. Prevents hallucinated APIs. |
| **Playwright** | Browser automation for testing UI. |
| **DeepWiki** | Documentation for any GitHub repo. |
