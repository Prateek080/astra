# Astra 🏹 (अस्त्र)

**A plugin for the complete development lifecycle.**

Works with [Claude Code](https://claude.ai/code), [Cursor](https://cursor.com), and as a standalone Python CLI. Same agents, skills, and methodology — three invocation paths.

```
Idea → Spec → Design → Plan → Architect → Implement → Review → Ship
```

**Two ways to automate:**
```bash
# Inside Claude Code or Cursor (markdown orchestrator):
/astra:forge "your feature description"

# Standalone Python CLI (SDK orchestrator):
astra-forge "your feature description"
```

Interrupted? Run `/astra:forge` again — it reads `.astra-cache/forge-state.json` and resumes from the last completed step.

Cross-project. Stack-agnostic. Works in any codebase.

## Prerequisites

| Requirement | Why |
|---|---|
| [Claude Code](https://claude.ai/code) or [Cursor](https://cursor.com) | Astra works as a plugin for both |

| Variable | Purpose |

| `SLACK_WEBHOOK_URL` | Slack pipeline notifications |

## Installation

One command — auto-detects your editor and installs the Python orchestrator:

```bash
curl -fsSL https://raw.githubusercontent.com/Prateek080/astra/main/install.sh | bash
```

The installer:
1. Clones/updates Astra to `~/astra`
2. Configures Claude Code (marketplace + plugin install) and/or Cursor (file copy)
3. Installs `astra-forge` CLI via pipx/pip (if Python 3.10+ found)
4. Reports token configuration status

<details>
<summary><b>Install for a specific editor</b></summary>

```bash
# Claude Code only
curl -fsSL https://raw.githubusercontent.com/Prateek080/astra/main/install.sh | bash -s -- --claude

# Cursor only
curl -fsSL https://raw.githubusercontent.com/Prateek080/astra/main/install.sh | bash -s -- --cursor
```

</details>

<details>
<summary><b>Manual installation</b></summary>

```bash
git clone https://github.com/Prateek080/astra.git ~/astra

# Claude Code — add to your shell config:
alias claude="claude --plugin-dir ~/astra"

# Cursor — run the install script:
bash ~/astra/scripts/install-cursor.sh

# Python orchestrator (optional):
pip install ~/astra
```

</details>

<details>
<summary><b>Update & Uninstall</b></summary>

```bash
# Update to latest
cd ~/astra && git pull && bash install.sh

# Uninstall (keep source files)
bash ~/astra/install.sh --uninstall

# Uninstall + delete everything
bash ~/astra/install.sh --purge
```

</details>

**Verify:** Run `/help` inside your editor — you should see all `/astra:*` commands. Run `astra-forge --help` for the SDK CLI.

> **For teams:** Each developer installs Astra and picks their editor. The project-specific files generated by `/astra:init` (`CLAUDE.md`, `.claude/rules/`) get committed to git — that's how the team shares project context.

## Setup

### `/astra:setup` — Configure your machine *(run once)*

```
/astra:setup
```

Configures your global environment with stack-agnostic coding principles and best practices:

| What it configures | Why |
|---|---|
| `~/.claude/settings.json` | Sets effort to high + auto-compact at 80% |
| `~/.claude/CLAUDE.md` | Personal coding standards — loaded into every session |
| Permissions, sandbox, statusline | Walks you through these step by step |

### `/astra:init` — Set up your project *(run once per codebase)*

```
/astra:init
```

Scans your project, asks 2 clarifying questions, then generates:

| What it generates | Purpose |
|---|---|
| `CLAUDE.md` | Project summary — stack, structure, dev commands, conventions |
| `.claude/rules/backend.md` | Backend conventions referencing your actual files |
| `.claude/rules/frontend.md` | Frontend conventions referencing your actual files |

### What files does Astra create?

| File | Created by | Commit? |
|---|---|---|
| `CLAUDE.md` | `/astra:init` | **Yes** |
| `.claude/rules/*.md` | `/astra:init` | **Yes** |
| `PRODUCT.md` | `/astra:forge` | **Yes** — living product context |
| `SPEC.md` | `/astra:spec` or `/astra:forge` | Optional |
| `DESIGN.md` | `/astra:design` or `/astra:forge` | Optional |
| `PLAN.md` | `/astra:plan` or `/astra:forge` | Optional |
| `TECHNICAL.md` | `/astra:architect` or `/astra:forge` | Optional |
| `docs/specs/*.md` | Auto-archived | Optional |
| `docs/designs/*.md` | Auto-archived | Optional |
| `docs/technical/*.md` | Auto-archived | Optional |
| `docs/plans/*.md` | Auto-archived | Optional |
| `docs/solutions/*.md` | `/astra:compound` | Optional |
| `docs/.agent-memory/*.md` | Agents (auto) | **No** — add to `.gitignore` |
| `graphify-out/` | `/graphify` (bundled skill) | **No** — persists across runs, gitignored |
| `.astra-cache/` | Pipeline (auto) | **No** — temporary, includes `forge-state.json` for resume |

## Choosing the Right Workflow

| Size | Example | Workflow |
|---|---|---|
| **Tiny** | Typo, rename, config change | Just ask Claude directly |
| **Small** | Single-file fix, add a field | Just ask Claude → optionally `review` |
| **Medium** | Feature across 2-5 files | `plan` → `implement` → `review` → `ship` |
| **Large** | New feature, unclear scope | `spec` → `design` → `plan` → `architect` → `implement` → `review` → `ship` |
| **End-to-End** | Full automation | **`forge "feature"`** or **`astra-forge "feature"`** |
| **Cleanup** | Refactor, pay debt | `debt-audit` → `plan` → `implement` → `review` → `ship` |

## Two Orchestration Paths

### Path 1: `/astra:forge` (Markdown — inside Claude Code/Cursor)

```
/astra:forge "add notifications system"
/astra:forge "add dark mode" --lite
/astra:forge "payment integration" --interactive
```

LLM-interpreted pipeline. Stages auto-proceed (zero-stop mode) — only stops on failures. Uses stage-gate skill for validation.

### Path 2: `astra-forge` (Python SDK — standalone CLI)

```bash
astra-forge "add notifications system"
astra-forge "add dark mode" --lite
astra-forge "payment integration" --interactive
astra-forge "auth system" --dry-run
```

Python-driven pipeline via Claude Agent SDK. Deterministic control flow, programmatic stage-gate checks, auto-permissions via hooks, Git automation, Slack notifications, checkpoint/resume.

### Both paths share:
- Same 7 agents (PM, designer, architect, planner, implementer, reviewer, debugger)
- Same 12 skills (methodology files + knowledge graph)
- Same artifacts (SPEC.md, DESIGN.md, PLAN.md, TECHNICAL.md)
- Same stage-gate checks (S1-S5, D1-D5, P1-P4, T1-T5, I1-I4)
- Same traceability: R{n} → D-R{n}/T-R{n} → Phase → Test

### SDK-only features:
| Feature | What it does |
|---|---|
| **Hooks** | Auto-permissions (zero popups), audit trail, context injection, readonly enforcement |
| **Git automation** | Auto-branch, phase commits, PR with R{n} checklist |
| **Slack notifications** | Stage pass/fail, pipeline completion alerts |
| **Agent memory** | Cross-session learning per agent persona |
| **Checkpoint/resume** | Pipeline state in `.astra-cache/pipeline-state.json` |
| **Scheduled tasks** | Nightly builds, PR review, weekly debt audit |
| **Deterministic checks** | Stage gates are Python regex/counting — not LLM judgment |

## Command Reference

### `/astra:forge` — Automated pipeline

```
/astra:forge "user auth with Google and GitHub OAuth"
/astra:forge "add dark mode" --lite
/astra:forge "payment integration" --interactive
```

Pipeline with built-in speed optimizations:

1. **Codebase scan** — cached for all agents
2. **PM agent** → `SPEC.md` → stage gate (S1-S5) → auto-proceeds
3. **Designer + Planner in parallel** → `DESIGN.md` + `PLAN.md` → stage gates (D1-D5, P1-P4)
4. **Architect agent** → `TECHNICAL.md` → stage gate (T1-T5)
5. **Implementer** → phase by phase → stage gate per phase (I1-I4)
6. **Reviewer** → findings → fix criticals
7. **Wrap-up** → archive artifacts, update PRODUCT.md

| Mode | When | Pipeline |
|---|---|---|
| **Full** (default) | 4+ requirements or full-stack | PM → Designer ‖ Planner → Architect → Implement → Review |
| **Lite** (auto/`--lite`) | ≤3 requirements, single type | PM → Planner → Implement → Review |
| **`--interactive`** | Manual approval gates | Same pipeline, pauses between stages |

---

### `/astra:spec` — Define what to build

```
/astra:spec "user auth with Google and GitHub OAuth"
```

PM subagent with pm-framework skill. RICE prioritization, Given/When/Then acceptance criteria, numbered requirements (R1, R2...).

### `/astra:design` — Create UI/UX design

```
/astra:design
```

Designer subagent with design-system skill. User journeys, component specs with exact tokens, states, accessibility, responsive behavior. Traces: D-R{n} → R{n}.

### `/astra:architect` — Create technical design

```
/astra:architect
```

Architect subagent with technical-architecture skill. API contracts, data models, ADRs, error handling, security. Traces: T-R{n} → R{n}.

### `/astra:plan` — Break work into phases

```
/astra:plan
```

Planner subagent explores codebase, creates phased plan with test gates. Parallel phases flagged.

### `/astra:implement` — Build phase by phase

```
/astra:implement
```

Executes plan phases. Test gate per phase. Stops after 2 failures.

### `/astra:review` — Catch issues

```
/astra:review
```

Reviewer subagent with 50+ item checklist. Critical / Warning / Suggestion findings.

### `/astra:ship` — Commit and PR

```
/astra:ship
```

Pre-flight checks (tests, lint, secrets, debug artifacts), commit, push, PR via `gh`.

### `/astra:debug` — Root cause analysis

```
/astra:debug "TypeError: cannot read property 'id' of undefined"
```

Debugger subagent. Hypotheses → systematic testing → minimal fix → regression test.

### Additional skills

| Skill | What it does |
|---|---|
| `/astra:compound` | Document solved problems to `docs/solutions/` |
| `/astra:debt-audit` | Scan for dead code, orphaned TODOs, missing tests |
| `/astra:workflow-guide` | Usage guidance for different task sizes |
| `/astra:retrospective` | Post-feature process assessment |

## Scheduled Tasks (SDK only)

```bash
# List tasks
astra-forge schedule list

# Create nightly forge run
astra-forge schedule create nightly-forge --cron "0 2 * * *"

# Create weekly debt audit
astra-forge schedule create weekly-debt --cron "0 9 * * 1"

# Run a task now
astra-forge schedule run nightly-forge
```

Tasks install as crontab entries. For nightly builds, create a backlog:

```markdown
<!-- .astra-cache/backlog.md -->
- [ ] add user notifications
- [ ] implement search functionality
- [ ] add dark mode toggle
```

The nightly task picks the next unchecked item and runs the full pipeline.

## Context Management

| Situation | Claude Code | Cursor |
|---|---|---|
| Finished a task | `/clear` | Start a new chat |
| Between implementation phases | `/clear` | Start a new chat |
| Long session, can't clear yet | `/compact` (at ~50%) | New chat when heavy |

**Built-in protections:**
- Commands delegate heavy exploration to subagents — separate context, not yours
- Auto-compaction at 80% via `/astra:setup`
- SDK path: pipeline checkpoint/resume survives session restarts

## Under the Hood

### Agents

| Agent | Role | Skill | Tools | Memory |
|---|---|---|---|---|
| **pm** | Requirements, PRD | pm-framework | Read-only | Yes |
| **designer** | UI/UX specs, components | design-system | Read-only | Yes |
| **architect** | API contracts, data models | technical-architecture | Read-only | Yes |
| **planner** | Codebase exploration, plans | plan-template | Read-only | Yes |
| **implementer** | Code execution, tests | implementation-patterns | Read + Write | Yes |
| **reviewer** | 50+ item review | review-checklist | Read-only | Yes |
| **debugger** | Root cause analysis | debugging-methodology | Read + Write | Yes |

### Stage-Gate Checks

Automated quality gates between pipeline stages. In the SDK path, these are deterministic Python functions (no LLM calls):

| Gate | Checks | What it catches |
|---|---|---|
| **Spec** (S1-S5) | Relevance, specificity, criteria, RICE, scope | Vague requirements, missing acceptance criteria, scope creep |
| **Design** (D1-D5) | R→D coverage, tokens, states, accessibility, orphans | Missing design elements, raw values, incomplete states |
| **Plan** (P1-P4) | R→phase coverage, test gates, task count, dependencies | Unmapped requirements, missing gates, oversized phases |
| **Technical** (T1-T5) | R→T coverage, API completeness, models, errors, conflicts | Incomplete schemas, missing error handling, route conflicts |
| **Phase** (I1-I4) | Tests, lint, types, criteria spot-check | Failing tests, lint errors, type errors |

### Hooks (SDK + Plugin)

| Hook | When | What it does |
|---|---|---|
| `SubagentStart` | Agent spawns | Injects codebase context cache |
| `PostToolUse` (Write/Edit) | File written | Appends to audit trail (`.astra-cache/audit.jsonl`) |
| `PreToolUse` (Write/Edit) | File about to be written | Enforces readonly for non-implementer agents |

### MCP Servers

| Server | Purpose |
|---|---|
| **Context7** | Up-to-date library docs |
| **Playwright** | Browser automation for UI testing |
| **DeepWiki** | Documentation for any GitHub repo |
| **GitHub** | Auto-PR, issue linking (requires `GITHUB_TOKEN`) |

### Path-Scoped Rules

Template rules in `rules/` are copied to `.claude/rules/` during `/astra:init`:

| Rule | Applies to | Key guidelines |
|---|---|---|
| `frontend.md` | `src/components/**` | Use component library, follow CSS approach, responsive, accessible |
| `api.md` | `src/api/**` | Follow route patterns, validate input, error envelope, integration tests |
| `tests.md` | `**/*.test.*` | Behavior tests, specific assertions, edge cases, independent |
